🧭 Decisions 3

Single Save-all button, sticky at the bottom

One submit button (no top duplicate). Made it position:sticky at the bottom of the form so it stays reachable on a long match list without scrolling to the very end. Disabled when there are zero unlocked matches.

Why: Plan offered an optional sticky duplicate at the top; a single sticky-bottom button is simpler and better UX for a long list. Locked matches' inputs are disabled (not submitted); the button is the only control.

Extract parse/filter into pure buildPredictionEntries()

New src/lib/predictions.ts holds buildPredictionEntries(rawPairs) returning validated {matchId, homeGoals, awayGoals}[]; the action reads FormData into raw pairs then calls it. Covered by src/lib/predictions.test.mts.

Why: Satisfies the 'behavioral change needs tests' rule with a pure, testable unit; keeps the action thin.

LocalKickoff client component with SSR-stable fallback

New 'use client' component renders <time> with the server (UTC) format as initial state, then reformats in useEffect using the browser timezone. suppressHydrationWarning avoids a mismatch warning.

Why: Only way Intl can pick up the client's timezone; keeps SSR deterministic. Stored data stays UTC; only display changes. Lock/scoring compare absolute instants, unaffected.

↪️ Deviations from spec 1

Locked matches show a 'Locked' label instead of a disabled button

Per-match Save buttons are gone (issue #5). For locked matches the score inputs are disabled and a small 'Locked' text marker is shown in their place, so the locked state stays visible.

Why: Preserves the prior visual signal that a match is locked now that the per-row button (which previously read 'Locked') no longer exists.

⚖️ Tradeoffs 2

Full RankingChart (all lines) per participant card

Each expandable card renders the complete by-match chart emphasizing that participant, rather than a single-line chart. Charts sit inside collapsed <details> so DOM cost is low.

#6 expanded-chart not exercised with live data

No group in the connected Supabase has any completed match (all WC2026 fixtures are future-dated), so the Performance section is empty everywhere and the new <details> charts have no data to render. Match results are GLOBAL (shared by all groups, read by production), so seeding a fake result to populate the chart would inject it into a real 19-member pool — not safe. Verified instead: build/typecheck/lint pass, the empty-state path renders cleanly without crashing, and the chart itself is the unchanged, already-tested RankingChart component (only new inputs are currentUserId=participant and a <details> wrapper).
