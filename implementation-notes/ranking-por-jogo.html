1. Design decisions

Rank = positional (1,2,3…), not competition ranking

The existing leaderboard renders index + 1 after ordering by total desc → exact desc → winner desc → joined_at asc. To "match the leaderboard" exactly, buildRanking assigns rank by sorted position (ties broken fully by the same comparator, with user_id as a final deterministic tiebreak). No shared ranks.

Per-day grouping uses the UTC calendar date

kickoff_utc is sliced to YYYY-MM-DD in UTC. This keeps the pure module deterministic and trivially testable (no timezone dependency). Tradeoff noted below.

Carry-forward at 0

A member with no prediction (or no score row) for a completed match contributes 0 points for that step but still appears in every standings snapshot, so lines are continuous.

rankDelta sign convention

rankDelta = previousRank − currentRank. Positive ⇒ moved up toward 1st ⇒ ▲. First step has delta 0.

2. Deviations from the spec

Column names: home_team_name / away_team_name

The plan's data-layer sketch said fetch home_team, away_team. The actual matches table (migration 001) has home_team_name / away_team_name. Using the real columns.

Members source = getLeaderboard reuse

Instead of a separate members query, getMatchRankingData reuses getLeaderboard(groupId) (which already returns every group member with user_id, display_name, avatar_url, joined_at via the group_memberships JOIN profiles shape). Avoids a duplicate query; the pure module only needs that subset.

Completed-matches set = matches with a match_results row

A match with a result but zero predictions won't appear in match_prediction_scores. To still walk it (everyone at 0), the data layer derives the completed set from match_results and filters the matches metadata, rather than from the scores view alone.

3. Tradeoffs considered

UTC vs local day grouping

Local (e.g. America/Sao_Paulo) day grouping matches user intuition better but makes the pure module timezone-dependent and harder to test deterministically. Chose UTC for the model; the page formats day labels for the locale at render time. open question

Chart: inline SVG vs Chart.js

Plan mandates hand-rolled SVG to stay server-component-first (no client JS) and match the custom-CSS design system. Accepted: more layout code, but zero new deps and SSR-friendly.

4. Open questions
  • Per-day grouping in UTC — acceptable, or should it follow a fixed tournament timezone?
  • "Worst match points" — computed over matches the player actually predicted (excludes carry-forward 0s). Confirm this is the intended meaning.
